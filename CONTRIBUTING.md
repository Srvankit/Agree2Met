# 🤝 Contributing to Agree2Met

Welcome to **Agree2Met**!
To maintain a clean, scalable, and professional codebase, all contributors must follow the guidelines below.

---

## 🚀 Workflow Overview

1. Clone the repository
2. Create your own branch
3. Make changes
4. Push your branch
5. Create a Pull Request (PR)
6. Get it reviewed
7. Merge after approval

---

## 🌿 Branching Rules

* Always create a new branch from `main`
* Use proper naming conventions:

```
feature/<feature-name>
bugfix/<bug-name>
hotfix/<urgent-fix>
```

### Examples:

* `feature/landing-page`
* `feature/dashboard-ui`
* `bugfix/navbar-alignment`

---

## 🔁 Pull Request Rules

* Every change must go through a PR
* Do NOT push directly to `main`
* Add a clear PR title and description
* Assign the admin as reviewer
* Keep PRs small and focused

---

## 🧠 Commit Message Guidelines

Use this format:

```
type: short description
```

### Types:

* `feat` → New feature
* `fix` → Bug fix
* `refactor` → Code improvement
* `style` → UI/Design changes
* `docs` → Documentation
* `chore` → Minor changes

### Example:

```
feat: add landing page hero section
fix: resolve navbar alignment issue
```

---

## 👀 Code Review Guidelines

Before approving any PR, check:

* Code readability
* Proper naming conventions
* No unnecessary code
* No console errors
* UI consistency
* Correct folder structure

---

## 💬 Review Etiquette

* Be constructive and respectful
* Suggest improvements clearly
* Use comments for discussion
* Request changes if necessary

---

## 🚫 Strict Rules

* ❌ No direct push to `main`
* ❌ No large PRs (keep changes small)
* ❌ No vague commit messages (e.g., "update", "fix")
* ❌ Do not modify someone else's assigned module

---

## 🔄 Daily Workflow

```
git pull origin main
git checkout -b feature/<your-feature>
```

---

## 👑 Maintainer Role

The maintainer (admin):

* Reviews all PRs
* Ensures code quality
* Merges approved changes

---

## 🎯 Goal

To build a clean, scalable, and professional project while learning real-world development practices.

---

💡 *"Code is not done when it works — it's done when it's reviewed."*
